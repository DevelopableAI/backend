# AGENTS.md

## Purpose

This repository is a Python CLI code generator. It takes a Prisma schema,
optionally merges business rules, and generates:

- An Express + TypeScript REST API.
- An optional Python integration test suite.
- Optional version-control/bootstrap artifacts for GitHub publishing.
- Optional cloud deployment bootstrap + deployment state tracking.

Treat this file as the fast working guide for agents and contributors.
`CLAUDE.md` is the deeper architecture narrative; this file is the
execution-focused counterpart.

## High-Level Runtime Flow (`main.py`)

`main.py` is the orchestration boundary and should remain easy to reason about.

1. Parse CLI arguments.
2. Parse Prisma schema into a canonical `spec` using `PrismaParser`.
3. Optionally merge `schema.rules.yaml` constraints using
   `BusinessRulesParser`.
4. Collect env var values referenced in schema (`collect_env_values`).
5. Run `Developer` to generate the API output tree.
6. Optionally run `Tester` to generate Python integration tests.
7. Optionally run `VersionControl` to write infra files, init git, create repo,
   push `main`.
8. Optionally run `Deployment` (intended after VersionControl publish).
9. Print LLM usage summary if LLM generation was enabled.

### CLI Ownership Notes

- `main.py` owns interactive user prompts and stage ordering.
- Agent classes own implementation details for each stage.
- Keep cross-stage coupling explicit in method parameters (`spec`, `api_plan`).

## Codebase Structure (Detailed)

### `agents/`

- `agents/developer.py`
  - Planner-first API generation orchestrator.
  - Creates API plan via `core/planner.py`.
  - Delegates render/fill/write to `core/assembler.py`.
  - Returns `api_plan` consumed by Tester/VersionControl/Deployment.

- `agents/tester.py`
  - Generates Python integration suite from `core/test_planner.py` + `api_plan`.
  - Must stay aligned with route decisions from `core/planner.py`.

- `agents/version_control.py`
  - Generates infra plan via `core/vc_planner.py`.
  - Writes `.gitignore`, initializes git, commits, creates GitHub repo via REST,
    pushes `main`.
  - Repository description should remain specific to generated API and retain
    “Generated by Developable” phrasing.

- `agents/deployment.py`
  - Cloud deployment orchestrator.
  - Resolves provider + credentials, ensures Dockerfile, builds image, deploys,
    records state to `.developable/state.json`.
  - Handles local deployment traceability and post-deployment automation hooks.
  - Must avoid LLM calls (deployment path should be deterministic and low-cost).

### `core/`

- `core/parser.py`
  - Produces canonical shared `spec` structure.
  - Extracts entities, fields, relations, enums, datasource/env usage, llm
    hints, auth metadata.

- `core/rules_parser.py`
  - Overlays business constraints on top of `spec`.
  - Adds denied endpoints, llm restrictions, primary parent overrides.

- `core/planner.py`
  - Converts `spec` → concrete API file manifest + generation context.
  - Central authority for routes, ownership behavior, auth-specific wiring,
    validation task scaffolding.

- `core/test_planner.py`
  - Converts (`spec` + `api_plan`) → test file manifest.
  - High-coupling file: route behavior changes frequently require updates here.

- `core/vc_planner.py`
  - Infra file plan for generated output repository.
  - Currently plans Dockerfile, docker-compose, CI workflow.

- `core/assembler.py`
  - Jinja rendering + bounded LLM fill execution.
  - Regeneration safeguards around user-modified tracked files.
  - Optional Prettier formatting if available in generated project.

- `core/deployment_state.py`
  - Local deployment ledger (`.developable/state.json`).
  - Source of truth for deployed resources discovered/created by Deployment.
  - Enables low-cost local tracking without cloud-list API calls.

- `core/providers/`
  - Deployment provider abstraction + concrete implementations.
  - `base.py`: interface + shared tag conventions.
  - `aws.py`: ECS/ECR deployment flow.
  - `gcp.py`: Cloud Run deployment flow.
  - `heroku.py`: container registry + app release flow.
  - `__init__.py`: provider registry and resolver.

### `templates/`

- `templates/express/api/`: generated API shape, infra templates, workflows.
- `templates/tests/`: generated Python integration test templates.
- Keep planner assumptions and template placeholders synchronized.

### `prompts/`

- Prompt fragments used only for bounded sections.
- Do not expand bounded usage into whole-file generation.

## Architectural Invariants

- **Planner-driven generation first**: structural behavior belongs in planners,
  not ad hoc in agents/templates.
- **Bounded LLM usage only**: deterministic templates/plans should do most work.
- **Test suite mirrors API plan**: do not rediscover routes independently.
- **Regeneration must stay git-aware**: preserve user edits unless `--force`.
- **Deployment traceability must be local-first**: always update
  `.developable/state.json`.

---

## Deployment-Specific Guidance

- Deployment should run after VersionControl publish when using the full
  “engineer entity” path.
- Keep credential collection explicit and minimal.
- Prefer provider API/SDK/CLI operations; avoid any LLM calls.
- Record all deployable resources in local state with consistent tag metadata.
- When possible, provision DB + apply schema (`prisma db push`) before service
  rollout.
- Post-deploy, generated CI/CD should redeploy only after CI success on `main`.
- Any temporary test run against remote endpoint must avoid polluting committed
  test code/config.

## Practical Edit Strategy

- Schema interpretation change:
  - Inspect `core/parser.py`, `core/planner.py`, `core/test_planner.py`, and
    affected templates.

- Route/auth/ownership/nested-resource change:
  - Update planner logic first, then corresponding templates/tests.

- Infra/CI change:
  - Update `core/vc_planner.py` and workflow templates together.

- Deployment behavior change:
  - Update `agents/deployment.py`, provider adapter(s), and
    `core/deployment_state.py` schema expectations together.

- Doc cleanup:
  - Keep `README.md`, `AGENTS.md`, and `CLAUDE.md` consistent.
